# Звіт про виконану роботу
**Дата:** 2026-03-06
**Проєкт:** Scratch Learning (Фаза 0) - MVP для перевірки гіпотез

## Короткий огляд
Сьогоднішній день був присвячений завершенню логіки валідації ігрових завдань (Спринт 1) та розробці бекенд логіки Telegram-бота (початок Спринту 2).

## Деталі виконаних завдань

### 1. Валідація ігрових завдань (Tasks Logic)
- **Проблема:** Завдання рівня "Pro", які використовували безкінечний цикл `always`, не збігалися з цільовими координатами (`target.x`, `target.y`) при симуляції та викликали помилки під час перевірки.
- **Рішення / Реалізація:** 
  - Написано та відлагоджено скрипт `scripts/validate-tasks.js`.
  - Скрипт успішно "розуміє" логіку блоків `always`, проводячи перевірку попадання `target` хоча б в одну ітерацію циклу.
  - Налаштовано режими перевірки `--check` та автовиправлення координат `--fix`.
- **Результат:** Всі 30 ігрових завдань (`data/tasks.json`) успішно пройшли 100% валідацію.

### 2. Завершення Спринту 1 & Підготовка до деплою
- **Проблема:** Для релізу frontend частини потрібен був конфіг для хостингу Vercel. Відсутність декількох структурних тек.
- **Рішення / Реалізація:**
  - Створено конфігураційний файл `vercel.json` у корені проєкту для маршрутизації API запитів (`/api/*` -> `server.js`) та статичних файлів.
  - Створено директорії `assets/characters/` та `audio/` згідно зі специфікацією ТЗ.
- **Результат:** Проєкт структурно готовий до деплою на безкоштовний тариф Vercel.

### 3. Розробка Telegram Бота (Спринт 2)
- **Проблема:** Необхідність механізму реєстрації дітей, видачі їм доступу та утримання уваги (retention).
- **Рішення / Реалізація:**
  - Ініціалізовано нове Node.js середовище в директорії `bot/` з використанням бібліотеки `telegraf` та `node-cron`.
  - Реалізовано команду `/start`: бот автоматично генерує унікальний `uid` для користувача, записує час реєстрації в локальну пам'ять (у майбутньому — БД) та видає персональне посилання на "Мапу пригод" (`progress.html`).
  - Реалізовано команду `/progress`: бот вираховує кількість днів з моменту реєстрації та повідомляє статистику.
  - Реалізовано 'Щоденну розсилку' (`node-cron`): кожного ранку о 9:00 (Київ) бот надсилає посилання на завдання дня всім зареєстрованим користувачам.
- **Результат:** Базовий функціонал MVP Telegram-бота успішно протестований локально (зокрема, команда тестування розсилки `/test_broadcast`).

## Обговорювані зауваження
Було піднято кілька критичних питань щодо готовності до закритого тестування:
1. **Спринт 1 не закритий остаточно:** Відсутній фактичний деплой фронтенду. Бот не може віддавати production-посилання `localhost:3000`.
2. **Ризик втрати даних Бота:** Зберігання In-memory (`usersDB`) призведе до втрати всіх підписників при першому ж рестарті процесу. Це неприпустимо для живого тестування (навіть закритого).
3. **Хостинг:** Усвідомлено архітектуру (Vercel для фронтенду, Railway/Render для Telegram-бота як background-процесу).

## Наступні кроки (Беклог)
**Критичні дії (Блокери перед запрошенням батьків у Sprint 2):**
1. Зробити дійсний деплой фронтенду на **Vercel** та отримати Production URL.
2. Впровадити збереження списку підписників бота у файл `bot/data/users.json` (або `json` fallback) замість чистої пам'яті.
3. Замінити `localhost:3000` в коді бота на отриманий Production URL Vercel-домену.

**Подальші плани (Sprint 3+):**
4. Деплой Telegram-бота на Railway / Render.
5. Підключення бази даних **Supabase** для повноцінного та безпечного збереження всього прогресу у хмарі.
