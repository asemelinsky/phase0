# Skill: daily-report

## Призначення
Призначений для автоматичного створення щоденного деталізованого звіту про роботу в проєкті наприкінці дня. Звіт формує список виконаних завдань, розв'язаних проблем та наступних кроків. Зберігається у файлі формату `YYYY-MM-DD.md` у директорії `reports/`.

## Коли використовувати
- Якщо користувач просить сформувати щоденний розгорнутий звіт.
- Наприкінці робочої сесії, щоб підсумувати досягнення.
- Коли користувач просить "Що ми сьогодні зробили?".

---

## Крок 1: Збір інформації (Context)
Перед генерацією звіту ознайомтеся з виконаними завданнями за допомогою таких інструментів:
1. Перегляньте чекліст виконаних дій, наприклад `task.md` або `implementation_plan.md` з історії артефактів.
2. Проаналізуйте історію сьогоднішньої бесіди або інших сесій.
3. Опціонально запустіть bash команду `git log --oneline --since="1 day ago"` щоб перевірити збережені коміти (якщо використовується git).

---

## Крок 2: Форматування звіту

Звіт повинен бути написаний **тільки українською мовою** та відповідати наступному шаблону:

```markdown
# Звіт про виконану роботу
**Дата:** YYYY-MM-DD
**Проєкт:** Scratch Learning (Фаза 0) - MVP для перевірки гіпотез

## Короткий огляд
[Тут 1-2 абзаци загальної інформації про головні досягнення дня і фокус спринту]

## Деталі виконаних завдань

### 1. [Назва великої фічі чи завдання]
- **Проблема / Що треба було зробити:** [Короткий опис, чому це розроблялося]
- **Рішення / Реалізація:** 
  - [Детально - які файли змінено, які модулі (напр. скрипт / бот) створено]
  - [Які інструменти чи бібліотеки використано (напр. mongoose / telegraf)]
- **Результат:** [Який кінцевий стан (тести проходять, сторінка завантажується, і т.д.)]

### 2. [Назва іншої великої фічі...]
...

## Наступні кроки (Беклог)
1. [Що ми плануємо зробити під час наступної сесії]
2. [Які технічні чи організаційні обставини слід врахувати (напр. створення БД, деплой)]
```

---

## Крок 3: Збереження Звіту

1. Перевірте, чи існує директорія `reports/` у корені проєкту. (Якщо ні — спершу створіть її через інструмент `write_to_file`).
2. Отримайте поточну дату системи у форматі `YYYY-MM-DD` (використовуючи поточний час або логічний контекст).
3. Збережіть згенерований звіт через `write_to_file`. Цільовий шлях: `reports/YYYY-MM-DD.md`.
4. Якщо такий файл вже існує за сьогоднішню дату — **доповніть** його, узгоджуючи нову інформацію з існуючим текстом (щоб не розбити структуру).
5. Повідомте користувача (`notify_user`), що звіт успішно створено, і надайте шлях до нього або короткий підсумок у чаті.
