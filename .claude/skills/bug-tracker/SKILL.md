## Операція 1: Додати баг

Створи запис через Notion MCP з полями:

| Поле | Значення |
|------|----------|
| `Назва` | `🐛 [короткий опис]` |
| `Тип` | `Task` |
| `Статус` | `To Do` |
| `Пріоритет` | див. таблицю нижче |
| `Спринт` | поточний активний спринт |
| `date:Дати:start` | сьогоднішня дата (YYYY-MM-DD) |
| `Acceptance Criteria` | файл/компонент + умови відтворення |

### Визначення пріоритету

| Пріоритет | Коли |
|-----------|------|
| `🔴 Критичний` | дитина не може виконати завдання / функціонал зламаний |
| `🟡 Середній` | є обхідний шлях, але заважає роботі |
| `🟢 Низький` | косметика, не впливає на навчання |

**Евристика для Кодомандри:** якщо баг у `lesson-engine`, `server.js`, перевірці блоків або TTS — швидше за все 🔴. Якщо UI/стилі/тексти — 🟢.

## Операція 2: Список відкритих задач

Запусти скрипт (використовує Notion REST API з точним фільтром):

```bash
node scripts/list-bugs.js              # всі To Do + In Progress
node scripts/list-bugs.js --critical   # тільки 🔴 Критичний
```

Або через npm: `npm run bugs` / `npm run bugs:critical`

> Потребує `NOTION_TOKEN` у файлі `.env` в корені проєкту.

## Операція 3: Взяти задачу в роботу (повний цикл)

### 3.1 Обрати задачу
Виконай Операцію 2, потім запитай: **"Яку задачу беремо? (номер або назва)"**

> **Пріоритет виконання:** завжди брати 🔴 Критичні першими. Якщо є хоча б одна критична задача — не братись за Середні або Низькі до її закриття.

Визначення пріоритету з картки:

| Пріоритет | Коли |
|-----------|------|
| `🔴 Критичний` | дитина не може виконати завдання / функціонал зламаний |
| `🟡 Середній` | є обхідний шлях, але заважає роботі |
| `🟢 Низький` | косметика, не впливає на навчання |

### 3.2 In Progress + читання коментарів
Оновити картку через Notion MCP: `Статус → In Progress`

**Обов'язково** прочитати всі коментарі картки через `notion-get-comments`, навіть якщо тіло картки здається порожнім — основна інформація про задачу може бути виключно в коментарях:
- Коментарі з префіксом `🤖 Claude |` — мої попередні звіти про виконання
- Коментарі без цього префіксу — зауваження від власника

**Як визначити поточне завдання:**
- Знайти останній коментар `🤖 Claude |` за часом
- Якщо після нього є коментарі від власника — **це і є оновлене завдання** (пріоритет над Acceptance Criteria картки)
- Якщо коментарів немає взагалі — виконувати Acceptance Criteria картки

**Коли зупинитися і запитати:**
- Якщо в картці або коментарях є слово "обговорити", "уточнити", "discuss" — розпочати діалог замість реалізації

Вивести у чат: список зауважень від власника які треба врахувати.

### 3.3 Виконати задачу
1. Врахувати `Acceptance Criteria` + всі зауваження з коментарів
2. Знайти пов'язані файли — читати код, не гадати
3. Внести виправлення
4. Якщо змінено `data/tasks.json` → запустити `node scripts/validate-tasks.js --check`

### 3.4 Commit
```bash
git add <змінені файли>
git commit -m "fix: [що виправлено]

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>"
```

### 3.5 Push + In Review + Коментар
```bash
git push
```
Після успішного push:
1. Оновити картку: `Статус → In Review`
2. Додати коментар до картки через Notion MCP (`notion-create-comment`) з коротким описом:
   - Що було зроблено (конкретно, технічно)
   - Які файли змінено
   - Як саме вирішено проблему

Формат коментаря (завжди починати з префіксу `🤖 Claude |` — щоб відрізнити від коментарів власника):
```
🤖 Claude | 🔧 Виправлено: [1-2 речення що саме зроблено]
📁 Файли: [список змінених файлів]
🔗 Commit: [перші 7 символів хешу]
```

Вивести підсумок:
```
✅ [Назва картки] → In Review
💬 Коментар додано до картки
🚀 Vercel деплоїть автоматично
```

> НЕ міняй статус на `Done` — це робить юзер після перевірки в браузері.
