# План впровадження підтримки декількох персонажів

Цей план описує технічні зміни, необхідні для підтримки кількох персонажів в одному завданні, кожен з яких має власний незалежний робочий простір коду та стан виконання.

## Пропоновані зміни

### [Компонент] Стан гри та інтерфейс (app/task.html)

#### [MODIFY] [task.html](file:///d:/aiscratch/claude/phase0/app/task.html)

- **Керування станом**:
    - Замінити поодинокі `charPos` та логіку персонажа `currentTask` на `characterManager`.
    - `characterManager` буде керувати масивом об'єктів `characterObjects`, кожен з яких містить:
        - `id`: Унікальний ідентифікатор.
        - `type`: 'cat', 'dog', 'parrot' тощо.
        - `pos`: `{x, y}` поточні координати.
        - `startPos`: `{x, y}` початкові координати для скидання.
        - `workspaceXml`: Збережений XML Blockly для цього конкретного персонажа.
        - `isActive`: Логічне значення для відстеження того, якого персонажа зараз редагують.

- **Покращення інтерфейсу**:
    - Додати контейнер `character-deck` над редактором коду або ігровим полем.
    - Впровадити "Картки персонажів":
        - Відображення емодзі/іконки персонажа.
        - Індикатор "Має код" (якщо робочий простір не порожній).
        - Підсвічування активного стану.
    - **Логіка перемикання**:
        - При натисканні на картку:
            1. Зберегти поточний робочий простір у `activeCharacter.workspaceXml`.
            2. Переключити посилання на `activeCharacter`.
            3. Очистити робочий простір і завантажити `newActiveCharacter.workspaceXml`.
            4. Оновити інтерфейс для фокусування на новому персонажі.

- **Рефакторинг рушія виконання**:
    - **`runCode()`**:
        - Замість того, щоб запускати лише поточний робочий простір, він має ініціювати виконання для всіх персонажів, для яких визначено код.
        - Використовувати `Promise.all()` для паралельного або послідовного руху персонажів, залежно від дизайну гри.
    - **`executeChain(character, startBlock)`**:
        - Передавати конкретний об'єкт персонажа в цю функцію.
        - Усі операції (наприклад, `move`, `jump`, `checkCollision`) будуть працювати над `pos` та `type` переданого персонажа.
    - **`draw()`**:
        - Перебирати всіх персонажів у менеджері та відображати їх на полотні.

- **Взаємодія з полотном (Canvas)**:
    - Оновити обробник натискання на полотні, щоб визначити, на якого персонажа було натиснуто, і автоматично переключитися на його "робочий стіл".

## План перевірки

### Ручна перевірка
1.  **Тест перемикання**:
    - Натиснути на картку персонажа А -> Додати блок `move_right`.
    - Натиснути на картку персонажа Б -> Робочий простір має бути порожнім.
    - Натиснути знову на картку персонажа А -> Блок `move_right` має з'явитися знову.
2.  **Тест виконання**:
    - Налаштувати рух персонажа А вправо, а персонажа Б — стрибок.
    - Натиснути "Запустити" -> Обидва персонажі мають виконати свої дії одночасно або послідовно, як заплановано.
3.  **Видимість інтерфейсу**:
    - Переконатися, що всі персонажі правильно відображаються на полотні разом із їхніми відповідними цілями.
